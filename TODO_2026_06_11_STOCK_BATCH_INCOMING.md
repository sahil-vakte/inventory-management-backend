# TODO - Stock Batch Incoming Logs and Roll Labels - 2026-06-11

## Goal

Create a stock batch incoming workflow where warehouse users can add incoming product rolls in metres, keep a proper incoming stock log, update product stock in metres, and generate roll labels for Dymo.

## Business Rule

When a new incoming batch is created for a product SKU, the system must add the total incoming roll meterage to the existing stock in metres.

Example:

- Product SKU: `AB`
- Current stock: `50` mtr
- Incoming batch:
  - Roll 1: `100` mtr
  - Roll 2: `50` mtr
- Incoming total: `150` mtr
- Updated stock: `50 + 100 + 50 = 200` mtr

Stock field to update:

- `StockItem.available_stock_in_mtr`

## Required Models

### StockBatch

Purpose: store one incoming batch header.

Suggested fields:

- `batch_id` - unique batch reference, generated by system
- `stock_item` - FK to `StockItem`
- `sku` - denormalized SKU for audit/reference
- `product_name` - denormalized product name for label/reporting
- `supplier` - supplier name or FK if supplier model exists later
- `created_by` - FK/user reference from logged-in session
- `batch_date` - system date by default
- `total_meterage` - sum of all roll meterage
- `roll_count`
- `notes`
- `is_deleted`
- `deleted_at`
- `created_at`
- `updated_at`

### StockBatchRoll

Purpose: store each roll inside an incoming batch.

Suggested fields:

- `batch` - FK to `StockBatch`
- `roll_number`
- `meterage`
- `label_generated`
- `label_generated_at`
- `label_generated_by`
- `created_at`
- `updated_at`

Constraints:

- Unique roll number per batch: `batch + roll_number`
- Meterage must be greater than `0`

### StockMovement

Use existing `StockMovement` table to log the stock increment.

Required movement values:

- `stock_item`
- `movement_type = IN`
- `quantity = total incoming meterage`
- `old_stock_level`
- `new_stock_level`
- `reference_number = batch_id`
- `reason = Incoming stock batch`
- `created_by`

## Required API Flow

### Create Incoming Batch

Endpoint:

- `POST /api/v1/stock-batches/`

Request example:

```json
{
  "sku": "AB",
  "supplier": "Supplier Name",
  "notes": "Optional notes",
  "rolls": [
    {
      "roll_number": 1,
      "meterage": 100
    },
    {
      "roll_number": 2,
      "meterage": 50
    }
  ]
}
```

Expected behavior:

- Validate SKU exists in `StockItem`
- Validate all roll meterage values are positive
- Create `StockBatch`
- Create all `StockBatchRoll` rows
- Calculate incoming total meterage
- Update `StockItem.available_stock_in_mtr`
- Create `StockMovement` incoming log
- Use `transaction.atomic()` so all changes save together or all rollback
- Do not create duplicate roll rows

## Stock Update Rules

- Stock quantity is stored in metres.
- Only update `StockItem.available_stock_in_mtr`.
- Incoming batch increases stock.
- Stock update must happen once per batch creation.
- Editing an existing batch needs special handling:
  - Either disallow editing rolls after stock update
  - Or calculate stock difference and create an adjustment movement
- Deleting a batch should not hard-delete stock history.
  - Prefer soft delete
  - If reversal is allowed, create a reversing `StockMovement`

## Duplicate Protection

Required:

- `batch_id` must be unique
- `batch + roll_number` must be unique
- API must not create duplicate batches on retry

Recommended idempotency option:

- Allow optional `external_reference` or `request_id`
- Unique constraint on `external_reference`
- If same request is retried, return existing batch instead of adding stock again

## Implementation Notes

- Use `transaction.atomic()` for batch create.
- Use row-level locking if possible:
  - `StockItem.objects.select_for_update().get(sku=sku)`
- Keep stock movement audit trail mandatory.
- Keep roll meterage as decimal if partial metres are possible; otherwise integer is acceptable.
- Do not update product `stock_value`; update `StockItem.available_stock_in_mtr`.
- Existing stock APIs must continue working with `available_stock_in_mtr`.
